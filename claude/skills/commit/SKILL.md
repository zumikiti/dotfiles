# Gitコミットガイド

## コミットタイミング

複数ステップのタスクを実行する際は、以下のタイミングで適宜コミットする:

- 1つの論理的な変更が完了した時（機能追加、バグ修正、リファクタリングなど）
- 次のステップに進む前に、現在の変更を確定させたい時
- テストが通る状態になった時

**やってはいけないこと:**
- すべての作業が終わってから一括コミット
- 動作しない中途半端な状態でのコミット

## コミット手順

### 1. 状態の確認

以下を並列で実行:
- `git status -u` で変更ファイルを確認（`-uall` は使わない）
- `git diff` と `git diff --cached` で差分を確認
- `git log --format="%s" -5` で直近5件のコミットメッセージのスタイルを確認

### 2. ステージング

- 関連するファイルのみを個別に `git add` する
- `git add -A` や `git add .` は使用しない
- `.env`、認証情報、`./claude/` ディレクトリ配下のファイルは絶対にステージしない

### 3. コミットメッセージ

直近のコミットメッセージのスタイルに倣い、以下のルールで作成する:

- **日本語**で記述する
- Conventional Commits のプレフィックスを付ける: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`, `modify:`, `test:` 等
- プレフィックスの後に半角スペースを1つ入れる
- 「何をしたか」を簡潔に書く（1行、50文字目安）
- コミットメッセージはHEREDOCで渡す

### 4. pre-commit hookが失敗した場合

- 失敗原因を修正する
- `--amend` は使わず、**新しいコミット**として再度コミットする

## コミット粒度の判断基準

**「1つの論理的な変更」ごとにコミットする。**
判断基準は「この変更だけを取り出して適用しても動作が成り立つか」という観点。

### 良い粒度の例

- ボタンのデザイン変更
- パスワードのバリデーション追加
- 不要なログ削除
- ユーザー登録のバックエンド実装（フロントエンドとは分ける）

### 避けるべきパターン

- **細かすぎる**: 変数名1つの変更、スペルミス修正だけで1コミット → 履歴が散らばり追跡困難
- **大きすぎる**: 「ログイン機能を実装」で新規ファイル・バグ修正・リファクタリング・設定変更をまとめる → レビュー困難

### 追加ルール

- 粒度は**タスク単位**で分ける。ファイル数ではなく論理的なまとまりで判断する
- 1コミットで数百行を超える変更になる場合は、さらに分割できないか検討する（テストファイルの新規作成は例外として許容）
- コミット前のテスト実行は必須ではない
- コミットはユーザーに確認せず自動で行ってよい。ただし、リポジトリの設定（pre-commit hook等）には従うこと
